## 3. API mới / thay đổi

### Endpoint canonical mới

- `GET /api/analytics/need-review-conversations`
- `GET /api/analytics/need-review-keywords`

### Endpoint legacy giữ lại

- `GET /api/analytics/negative-conversations`

Endpoint này vẫn hoạt động nhưng được đánh dấu `meta.endpointStatus = legacy`.

### Keyword modes

- `GET /api/analytics/negative-keywords?mode=negative`
- `GET /api/analytics/negative-keywords?mode=needReview`
- `GET /api/analytics/negative-keywords?mode=issue`
- `GET /api/analytics/need-review-keywords`

Logic:

- `negative`: chỉ `sentimentLabel = 'negative'`.
- `needReview`: `needStaffReview = 1 OR issueFlag = 1 OR sentimentLabel = 'negative'`.
- `issue`: `issueFlag = 1 OR issueType IS NOT NULL`.

Nếu DB chưa có issue columns, issue mode trả rỗng an toàn thay vì lỗi.

## 7. Rủi ro còn lại

- Issue metadata chưa tồn tại trong DB production cho đến khi migration được duyệt và chạy.
- Dữ liệu cũ chưa có `issueFlag / issueType / issueReason / issueConfidence`, nên issue keyword mode hiện trả rỗng nếu schema chưa có cột.
- IssueDetector vẫn phụ thuộc static rules, cần monitoring typo/slang/context-required cases.
- `need-review-keywords` hiện vẫn phụ thuộc dữ liệu keyword đã lưu; sau migration và reprocess có thể phong phú hơn.
- Một số màn AI/demo vẫn là demo nhưng đã/đang được gắn nhãn rõ, không dùng làm bằng chứng production.

## 8. Khuyến nghị tiếp theo

1. Review và approve migration issue metadata.
2. Chạy migration sau khi có backup DB.
3. Chạy dry-run reprocess để so sánh `needStaffReview`, `issueType`, false positive/false negative.
4. Sau dry-run đạt ngưỡng, mới reprocess DB production.
5. Tạo fresh blind test mới sau khi đã thêm issue metadata.
6. Giữ ViSoBERT ở trạng thái experimental cho đến khi `/health` và benchmark xác nhận.
